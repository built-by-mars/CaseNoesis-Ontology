# Change Proposals

This directory contains locally-drafted change proposals for the CASE and UCO ontologies, and for supported extension ontologies such as the [CAC Ontology](https://github.com/Project-VIC-International/CAC-Ontology).

Each file is a filled-in version of the official [change proposal template](../docs/templates/change-proposal-template.md), generated by the SDK's `draft_change_proposal` MCP tool or written manually following the [change-proposal recipe](../docs/recipes/change-proposal.md).

## Workflow

1. A gap is identified while modeling data with the SDK — either manually or when `validate_graph` rejects a graph with `undeclared_concepts` (strict concept coverage)
2. Existing proposals are checked in the [UCO](https://github.com/ucoProject/UCO/issues), [CASE](https://github.com/casework/CASE/issues), and [CAC](https://github.com/Project-VIC-International/CAC-Ontology/issues) issue trackers (`check_existing_proposals` searches all three)
3. A proposal is drafted here using the official template
4. The developer reviews, refines, and submits the proposal as a GitHub issue in the appropriate repository
5. For terms needed immediately, the concept is declared in a local extension file (e.g. `ontology/cac/local/cacontology-sdk-pending.ttl`) referencing the filed issue, so strict validation passes until the proposal lands upstream

## File naming

Files are named `{concept-slug}.md`, e.g., `drone-telemetry-facet.md`.

## Submitting

When ready, copy the contents of the proposal file into a new GitHub issue:

- **UCO concepts** (general cyber-observables, identities, actions): https://github.com/ucoProject/UCO/issues/new
- **CASE concepts** (investigation-specific roles, actions, metadata): https://github.com/casework/CASE/issues/new
- **CAC concepts** (crimes-against-children domain): https://github.com/Project-VIC-International/CAC-Ontology/issues/new

## Filed CASE proposals

Three companion proposals asking CASE to add jurisdiction-neutral **stub namespaces** for legal process and procedure, so application-domain ontologies (CAC Ontology, `cryptoinv`, e-discovery, insider threat) inherit shared legal concepts from CASE instead of forking them. They complement [CASE #189](https://github.com/casework/CASE/issues/189) (process meta-model / business rules namespaces) and the CASE 1.5.0 investigator-role work ([#178](https://github.com/casework/CASE/issues/178), [#183](https://github.com/casework/CASE/issues/183), [#187](https://github.com/casework/CASE/issues/187)).

| Proposal | Issue | Stub concepts |
|---|---|---|
| [`criminal-process-and-procedure-namespace-stub-concepts.md`](criminal-process-and-procedure-namespace-stub-concepts.md) | [CASE #192](https://github.com/casework/CASE/issues/192) | `CriminalCharge`, `Plea`, `CriminalProceeding`, `Sentence`, `ForfeitureOrder`, `RestitutionOrder` |
| [`civil-process-and-procedure-namespace-stub-concepts.md`](civil-process-and-procedure-namespace-stub-concepts.md) | [CASE #193](https://github.com/casework/CASE/issues/193) | `CivilAction`, `CivilProceeding`, `LegalHold`, `DiscoveryRequest`, `CivilJudgment`, `Settlement` |
| [`corporate-process-and-procedure-namespace-stub-concepts.md`](corporate-process-and-procedure-namespace-stub-concepts.md) | [CASE #194](https://github.com/casework/CASE/issues/194) | `CorporateInvestigationMandate`, `PolicyViolation`, `DisciplinaryOutcome`, `Referral`, `RemediationAction` |

Each proposal ships with a tested example graph (`.jsonld`), draft T-Box (`.ttl`), and SPARQL competency queries (`.sparql`); all three pass `make test-proposal PROPOSAL=<slug>` (`Conforms: True`). If the committee adopts the namespaces, `cryptoinv` and the CAC legal modules should be refactored to subclass the CASE stubs.

A fourth CASE proposal was appended as a comment to an existing community issue rather than filed as a new issue:

| Proposal | Issue | Concepts |
|---|---|---|
| [`case-number-and-related-investigation-representation.md`](case-number-and-related-investigation-representation.md) | [CASE #191 (comment)](https://github.com/casework/CASE/issues/191#issuecomment-4926701151) | `investigation:caseIdentifier` (subproperty of `uco-core:externalIdentifier`, answering the imaging-tool "Case number" field) and `investigation:InvestigationRelationship` (relating investigations that carry different case numbers across organizations, e.g. platform trust & safety → NCMEC CyberTip → state ICAC → local PD → state lab). Re-parents [CAC-Ontology #41](https://github.com/Project-VIC-International/CAC-Ontology/issues/41)'s local `caseNumber` when adopted. |

A follow-up [comment on CASE #192](https://github.com/casework/CASE/issues/192#issuecomment-4927385975) recommends modeling **conspiracy** (and attempt, solicitation, aiding-abetting, and § 924(c)-style derivative counts) as `CriminalCharge` instances with two optional properties — `offenseForm` (open vocabulary) and `objectOffense` (charge-to-charge link) — rather than as subclasses. The recommendation was validated against *U.S. v. Perry & O'Dell* (W.D. Mo. `2:22-cr-04065-BCW`, 45 counts spanning conspiracy, attempted murder, and fourteen § 924(c) counts); the SDK-local implementation is the [`legalproc` extension](../extensions/legalproc/) and the validated exemplar is `examples/pacer/wdmo_2022_cr_04065/`.
